Page 1

时间字段边界

高优先级

这里要明确 iter-1 到底抽哪些时间字段，以及时间抽取服务于什么业务目标。关键不是做完整日历，而是决定“现在还值不值得看”。

flowchart LR
  标题/摘要/正文 --> 角色识别
  角色识别 --> 开始时间
  角色识别 --> 结束时间
  角色识别 --> 截止时间
  开始时间 --> time_status
  结束时间 --> time_status
  截止时间 --> participation_status

Q1. iter-1 最小时间字段集合要到哪一步？

推荐方案。足够支撑时效治理、过期隐藏、主列表排序和可参与判定。

实现更轻，但会削弱 ongoing 和 expired 的判定精度，容易让规则长期半成品化。

更适合后续调规则和排障，但会把 iter-1 schema 一次扩得更大。

Q2. 当 `deadline_at` 和 `event_end_at` 同时存在时，谁优先决定“是否过期”？

推荐方案，更符合报名、申请、征集、投递类内容的真实决策逻辑。

规则统一，但会误把“活动未开始、报名已截止”的内容判成仍有效。

Page 2

可参与 / 不可参与判定

主列表入口规则

这里决定主列表到底是不是“机会流”。如果这个边界不清楚，后面的时间提取和排序都会变形。

flowchart TD
  命中回顾/名单/公示 --> non_participable
  命中报名/申请/招募 --> 候选机会
  候选机会 --> 检查时间
  检查时间 --> participable
  检查时间 --> uncertain
  检查时间 --> non_participable

Q3. `participation_status` 是否要成为正式字段？

推荐方案。这样主列表、筛选、统计、排序都能围绕一个明确字段展开。

实现更省，但会让规则分散在多个地方，后续更难解释和测试。

Q4. 默认主列表应优先显示哪一类内容？

推荐方案，符合“机会引擎”而不是“全文归档流”的定位。

会保留更多灰区内容，但主列表信噪比会下降。

Page 3

排序权重

规则排序

这里决定系统是不是继续按 `published_at desc` 伪装成“能发现机会”，还是正式升级为规则排序。

flowchart LR
  participation_status --> ranking_score
  primary_category --> ranking_score
  time_status --> ranking_score
  deadline proximity --> ranking_score
  freshness --> ranking_score
  source quality --> ranking_score

Q5. iter-1 排序公式要做到哪一步？

推荐方案。把机会优先逻辑显式化，避免“规则在脑子里，排序还在按时间”的假完成。

代码改动少，但难解释、难复盘，也不利于后续统计和调参。

Q6. 排序维度里谁应该拥有最高权重？

推荐方案。先回答“还能不能做”，再回答“急不急、值不值”。

实现简单，但会让新鲜的低价值内容压住稍旧但更重要的机会。

长期稳定性

Q7. iter-1 是否正式采用三层存储？

推荐方案。哪怕暂时仍在一个库里，也要先把角色边界明确下来。

开发更快，但后续排障、重跑规则、适配上游字段变化时会很吃亏。

Q8. 原文与摘要的主策略是什么？

推荐方案。摘要用于筛选和浏览，原文用于跳转和查证，不把全文 HTML 作为唯一业务输入。

短期最省事，但会把排序、分类、安全和存储问题绑死在一起。
